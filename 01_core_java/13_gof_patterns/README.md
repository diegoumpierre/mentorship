# GoF Design Patterns

| Activities |
|------------|
| - [ ] Singleton — implement it four ways: eager, lazy, holder idiom, enum |
| - [ ] Factory Method on a small product hierarchy |
| - [ ] Abstract Factory that returns matched families of products |
| - [ ] Builder with a fluent API and a final `build()` that validates |
| - [ ] Prototype using both `clone` and a copy constructor — pick one and justify it |
| - [ ] Adapter, both flavours: object adapter (composition) and class adapter (inheritance) |
| - [ ] Bridge — separate the abstraction from the implementation hierarchy |
| - [ ] Composite tree (folders/files is the boring-but-clear example) |
| - [ ] Decorator stacked at least three deep |
| - [ ] Facade in front of a noisy subsystem |
| - [ ] Flyweight where the shared part is genuinely heavy |
| - [ ] Proxy: virtual, protection, and a dynamic one via `java.lang.reflect.Proxy` |
| - [ ] Chain of Responsibility — handlers that may pass along or stop |
| - [ ] Command with `execute` and `undo` |
| - [ ] Interpreter for a tiny expression language |
| - [ ] Iterator: a custom one and one that just hooks into `Iterable` |
| - [ ] Mediator that decouples a handful of components |
| - [ ] Memento that stores and restores state without breaking encapsulation |
| - [ ] Observer with `PropertyChangeSupport` and a hand-rolled listener list |
| - [ ] State machine where transitions are real classes |
| - [ ] Strategy injected as a lambda |
| - [ ] Template Method with a couple of hooks for subclasses |
| - [ ] Visitor over a sealed hierarchy |
| - [ ] For each pattern, write a runnable example and one focused test that proves the intent |
| - [ ] Note where modern Java (lambdas, records, sealed types) makes the classic shape unnecessary |
